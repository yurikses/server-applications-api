# Практическая работа по FastAPI

В данном файле описаны результаты выполнения заданий 9.1, 10.1, 10.2, 11.1 и 11.2, а также инструкции по их проверке.

## Установка и запуск

1. Клонируйте репозиторий и перейдите в директорию:
   ```bash
   git clone https://github.com/yurikses/server-applications-api/tree/FourthPractice
   ```

2. Установите зависимости:
   ```bash
   pip install -r requirements.txt
   ```

3. Запустите приложение:
   ```bash
   uvicorn main:app --reload
   ```
   Интерактивная документация (Swagger UI) для проверки основной функциональности будет доступна по адресу: **http://127.0.0.1:8000/docs**.

## Проверка и тестирование ключевых сценариев

Проверка функциональности (создание, чтение, удаление) реализована через изолированные модульные тесты:
- Запуск синхронных тестов: `pytest tests.py -v`
- Запуск асинхронных тестов: `pytest async_tests/test.py -v`

2. Ручное тестирование через curl:
- Проверка кастомной ошибки 404:
  ```bash
  curl -X GET "http://127.0.0.1:8000/products/99"
  ```

  ```bash

       -H "Content-Type: application/json" \
       -d "{\"username\":\"Alex\", \"age\":15, \"email\":\"not-an-email\", \"password\":\"123\"}"
  ```

## Задача 10.1: Пользовательская обработка ошибок
Что было сделано:
- Созданы пользовательские классы исключений: `ItemNotFoundException` и `InsufficientStockException`.
- Написана Pydantic-модель `ErrorResponse` для консистентного вывода ошибок (содержит поля `error_code`, `message`, `details`).
- Зарегистрированы глобальные обработчики исключений `@app.exception_handler`.
- Созданы эндпоинты `/products/{product_id}` и `/products/{product_id}/buy`, которые выбрасывают эти ошибки при неверном ID товара или нехватке товара на складе.

Как проверить:
Запустите сервер (`uvicorn main:app --reload`) и выполните запросы (товар ID 1 есть, ID 99 нет):
1. Ошибка 404 (Товар не найден):
   ```bash
   curl -X GET "http://127.0.0.1:8000/products/99"
   ```
2. Ошибка 400 (Недостаточно товара):
   ```bash
   curl -X POST "http://127.0.0.1:8000/products/1/buy?amount=10"
   ```
Оба ответа вернут красивый JSON в формате модели `ErrorResponse`.



## Задача 10.2: Обработка ошибок валидации (RequestValidationError)
Что было сделано:
- Создана Pydantic-модель `UserProfile` с жесткой валидацией полей: `conint(gt=18)`, `EmailStr`, `constr(min_length=8, max_length=16)`.
- Реализован перехватчик стандартной ошибки FastAPI `@app.exception_handler(RequestValidationError)`.
- Теперь вместо стандартного нечитаемого массива ошибок приложение возвращает понятный структурированный ответ со статусом 422 Unprocessable Entity, где четко указано, в каком поле произошла ошибка.

Как проверить:
Отправьте невалидные данные на эндпоинт:
```bash
curl -X POST "http://127.0.0.1:8000/profiles" \
     -H "Content-Type: application/json" \
     -d "{\"username\":\"Alex\", \"age\":15, \"email\":\"not-an-email\", \"password\":\"123\"}"
```
В ответе будет подробно расписано, почему `age`, `email` и `password` не прошли проверку.

## Задача 11.2: Асинхронное тестирование (pytest-asyncio + Faker + ASGITransport)
Что было сделано:
- Написано асинхронное приложение `async_test/app.py` с корректными HTTP-статусами (включая 204 No Content для DELETE).
- В папке `async_tests/` создан файл `test.py`.
- Настроены асинхронные тесты с помощью `pytest-asyncio`.
- Запросы делаются напрямую к ASGI-интерфейсу с помощью `httpx.AsyncClient` и `ASGITransport`, минуя запуск реального сервера `uvicorn`.
- Для генерации реалистичных данных пользователей используется библиотека `Faker`.
- Состояние словаря очищается между тестами.
- Протестированы все сценарии: создание (201), получение (200), несуществующий юзер (404), удаление (204) и повторное удаление (404).

Как проверить:
Установите зависимости (`pip install pytest-asyncio httpx faker`), затем запустите асинхронные тесты:
```bash
pytest async_tests/test.py -v 
```
Все тесты должны успешно пройти (PASSED).
